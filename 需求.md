# OCR 現場補料單比對系統需求

## 專案目的

本專案用於處理紙本現場補料單。使用者透過 LINE OA 拍照或上傳圖片後，系統透過 OCR 與 Gemini AI 分析補料單內容，將辨識出的資料與 Notion 中既有品項主檔比對。

比對結果需要完整寫入 Notion，並回覆 LINE 使用者本次掃描摘要，例如總筆數、通過筆數、異常筆數。

核心原則：

- OCR 掃描出幾筆，就必須寫入 Notion 幾筆。
- 即使 Gemini 辨識不清楚、圖片文字模糊、品號查不到主檔，也必須產生紀錄。
- `check` 欄位依比對結果打勾或不打勾。
- 最後 LINE 回覆本次掃描總結。

## 系統組成

### LINE OA

LINE OA 是使用者入口。

負責：

- 接收使用者文字指令。
- 接收補料單圖片。
- 回覆操作提示。
- 回覆 OCR 與比對結果。

目前啟動關鍵字：

```text
掃描現場補料單
```

使用流程：

```text
使用者輸入：掃描現場補料單
系統回覆：已進入掃描模式，請直接拍照或從相簿上傳現場補料單圖片。
使用者上傳圖片
系統進行 OCR、主檔比對、寫入 Notion、回覆 LINE 摘要
```

注意：

- 不使用會重複觸發的 Quick Reply 按鈕作為掃描啟動按鈕，避免造成訊息回覆循環。
- LINE Quick Reply 不能強制開啟使用者相簿，只能文字引導。

### Gemini AI

Gemini 用於分析補料單圖片並輸出結構化資料。

目前建議使用 n8n 內建 Gemini 圖片分析節點，不再用 HTTP Request 手打 API。

建議設定：

```text
Resource：Image
Operation：Analyze Image
Model：models/gemini-2.5-flash 或可用的較新模型
Input Type：Binary File(s)
Input Data Field Name(s)：image
Simplify Output：開啟
```

Gemini 輸出應盡量要求 JSON 格式：

```json
{
  "order_no": "製令單號或單據號碼，無法判讀則空字串",
  "date": "YYYY-MM-DDTHH:mm:ss+08:00，無法判讀則空字串",
  "items": [
    {
      "part_no": "品號",
      "spec": "規格",
      "defective_qty": 0,
      "request_qty": 0,
      "received_qty": 0,
      "note": "備註或無法判讀原因"
    }
  ]
}
```

品號辨識原則：

- 目前先採用第一階段品號規則：`品號第一個字元一定是英文 A-Z`。
- Gemini 分析圖片時，若疑似品號開頭不是英文，應優先判斷是否 OCR 誤讀，例如 `0/O`、`1/I`、`5/S`、`8/B`。
- 若仍無法確認，`part_no` 保留空字串，並在 `note` 寫明「品號開頭非英文或無法確認」。
- 後續資料量變大後，會再補充更完整的品號命名規則。

如果 Gemini 無法辨識清楚，`格式轉換` 節點仍必須至少輸出一筆異常紀錄，避免流程中斷。

### Notion

Notion 負責資料管理與記錄。

#### 品項主檔

資料庫名稱：

```text
品項主檔-吹氣盒
```

資料庫 ID：

```text
33eb3ad1d1cd80fc9674e730c3c87469
```

目前欄位：

```text
品號
品名
規格
```

用途：

- 作為主檔規格比對來源。
- 目前只處理「吹氣盒」品項資料。

#### OCR 結果紀錄

資料庫名稱：

```text
Easan現場補料單
```

資料庫 ID：

```text
2f6b3ad1d1cd8068a597e0994b74654c
```

目前欄位：

```text
製令單號
製另單號 check
日期
品號
品號 check
規格
規格 check
不良數量
需領數量
實領數量
備註
```

用途：

- 儲存所有 OCR 掃描後的補料單明細。
- 每一筆 OCR 明細都必須建立一筆 Notion page。
- 即使比對異常或 OCR 不完整，也要寫入。

## n8n 流程

建議主線流程如下：

```text
LINE Webhook
→ 解析 LINE 事件
→ 是否為圖片
→ LINE 取得圖片內容
→ Gemini ai 分析
→ 格式轉換
→ 查詢品項主檔
→ 規格比對
→ 寫入 Notion
→ 彙整比對結果
→ 建立 LINE Flex 回覆
→ LINE 回覆 OCR 比對結果
→ Webhook 回應 OK
```

非圖片或未進入掃描模式分支：

```text
是否為圖片 false
→ 建立非圖片回覆
→ LINE 回覆非圖片提醒
→ Webhook 回應 OK
```

## 重要流程規則

### 不要在查詢品項時用 OCR 品號過濾

曾遇到問題：

```text
格式轉換：8 items
查詢品項：1 item
規格比對：1 item
寫入 Notion：1 item
```

原因是 Notion `Get many database pages` 使用 `品號 equals {{ $json["品號"] }}` filter 後，只會輸出查到的主檔資料，查不到的 OCR 明細會消失。

正確做法：

```text
查詢品項主檔
Resource：Database Page
Operation：Get Many
Database：品項主檔-吹氣盒
Return All：true
Simplify：false
Filter：清空，不設定
```

然後在 `規格比對` Code 節點中：

```text
OCR 明細 = $('格式轉換').all()
主檔資料 = $('查詢品項').all()
```

用 Code 自行依 `品號` 尋找主檔，確保 OCR 幾筆就輸出幾筆。

### 寫入 Notion 必須在彙整前

正確 item 數量應為：

```text
格式轉換：多筆，例如 8 items
查詢品項主檔：主檔多筆
規格比對：與格式轉換相同筆數，例如 8 items
寫入 Notion：與格式轉換相同筆數，例如 8 items
彙整比對結果：1 item
建立 LINE Flex 回覆：1 item
```

不要把 `寫入 Notion` 接在 `彙整比對結果` 後面，否則 Notion 只會寫入一筆總結資料。

## 格式轉換需求

`格式轉換` 節點負責把 Gemini 輸出轉成逐筆補料明細。

必須滿足：

- Gemini 回傳 JSON 時，展開 `items`。
- Gemini 回傳 markdown code block 時，移除 ```json 與 ```。
- Gemini 回傳非合法 JSON 時，仍輸出一筆異常紀錄。
- Gemini 沒有 `items` 時，仍輸出一筆異常紀錄。
- OCR 不清楚時，仍輸出空品號/空規格紀錄，備註寫明原因。
- 品號第一個字元必須是英文 A-Z；若不是，應標記為異常，不可當作正常品號通過比對。

輸出欄位應統一為：

```text
製令單號
日期
品號
規格
不良數量
需領數量
實領數量
備註
OCR狀態
```

若無法判讀：

```text
製令單號：未判讀
品號：空字串
規格：空字串
數量：0
備註：寫明 Gemini 未回傳、JSON 無效、OCR 未判讀品號等原因
OCR狀態：辨識異常 或 辨識不完整
```

品號第一碼檢查範例：

```javascript
function isValidPartNo(value) {
  return /^[A-Za-z]/.test(String(value || '').trim());
}
```

若 `isValidPartNo(partNo)` 為 false：

```text
品號：可保留 OCR 原始值或改為空字串，依當時流程設定
品號 check：false
規格 check：false
備註：品號開頭非英文或無法確認
OCR狀態：辨識不完整
```

## 規格比對需求

`規格比對` 節點負責產出 check 欄位與異常原因。

比對來源：

```text
OCR 明細：格式轉換節點輸出
主檔資料：查詢品項主檔節點輸出
```

比對規則：

```text
製另單號 check：製令單號存在且不是「未判讀」
品號 check：OCR 品號第一碼為英文，且 OCR 品號與主檔品號相同
規格 check：OCR 規格與主檔規格相同
```

判斷：

```text
全部 check 通過 → 比對結果 = OK
任一 check 不通過 → 比對結果 = 異常
```

備註應記錄：

- OCR 未判讀製令單號
- OCR 未判讀品號
- 品號開頭非英文或無法確認
- OCR 未判讀規格
- 主檔查無品號
- 規格不符，主檔規格為何
- Gemini/OCR 原始異常說明

比對時需 normalize 字串，避免空白、大小寫、標點造成誤判。

## 寫入 Notion 需求

`寫入 Notion` 節點必須接在 `規格比對` 後面。

欄位對應：

```text
製令單號        → {{ $json["製令單號"] }}
製另單號 check → {{ $json["製另單號 check"] }}
日期            → {{ $json["日期"] }}
品號            → {{ $json["品號"] }}
品號 check      → {{ $json["品號 check"] }}
規格            → {{ $json["規格"] }}
規格 check      → {{ $json["規格 check"] }}
不良數量        → {{ $json["不良數量"] }}
需領數量        → {{ $json["需領數量"] }}
實領數量        → {{ $json["實領數量"] }}
備註            → {{ $json["備註"] }}
```

Checkbox 欄位必須傳 boolean：

```text
true / false
```

不要傳字串：

```text
"true" / "false"
```

## 彙整比對結果需求

`彙整比對結果` 節點放在 `寫入 Notion` 後面。

輸出只需要 1 item，給 LINE Flex 回覆使用。

應包含：

```text
order_no
total
passed
abnormal
results
first_notion_url
```

統計規則：

```text
total = 本次 OCR 明細總筆數
passed = 比對結果為 OK 的筆數
abnormal = 比對結果不是 OK 的筆數
```

## LINE Flex 回覆需求

LINE 回覆需要總結本次掃描結果。

內容包含：

```text
OCR 比對完成
製令單號
總筆數
通過筆數
異常筆數
前幾筆明細與備註
```

若有 Notion URL，可提供查看 Notion 紀錄按鈕。

Flex 回覆 Code 必須回傳 n8n 格式：

```javascript
return [{
  json: {
    body
  }
}];
```

不能回傳：

```javascript
return body;
return { body };
```

## 已遇過的問題與排除方式

### Gemini API key undefined

錯誤：

```text
API key not valid
key=undefined
```

原因：

```text
GEMINI_API_KEY 環境變數沒有設定
```

目前建議改用 n8n Gemini 節點 credential，不再手打 HTTP Request。

### Gemini 模型不存在

錯誤：

```text
models/gemini-1.5-flash is not found for API version v1beta
```

處理：

```text
改用 gemini-2.0-flash、gemini-2.0-flash-lite 或 models/gemini-2.5-flash
```

實際可用模型以 n8n Gemini 節點下拉選單為準。

### Gemini 額度限制

錯誤：

```text
The service is receiving too many requests from you
Quota exceeded
```

處理：

- 等待錯誤訊息指定秒數後重試。
- 不要連續 Execute step。
- 使用較省模型。
- 啟用 Google Cloud / AI Studio 付費或提高 quota。

### LINE 圖片 binary 取不到

曾遇錯誤：

```text
未取得 LINE 圖片 binary data
```

原因：

```text
Code 節點使用 $binary.image 取值不穩
```

處理：

```javascript
const item = $input.first();
const binary = item.binary?.image;
```

### Notion 只寫入一筆

原因：

```text
寫入 Notion 接在彙整比對結果後面
或查詢品項使用 filter 導致只剩查到的 1 item
```

處理：

```text
寫入 Notion 必須接在規格比對後面
查詢品項主檔必須 Return All，不用 filter
規格比對用 Code 自行比對全部 OCR 明細與全部主檔
```

### Code 節點 Multiple matches

錯誤：

```text
Multiple matches
```

原因：

```javascript
$('格式轉換').item.json
```

在多筆 item 情境下 n8n 不知道要取哪一筆。

處理：

```javascript
$('格式轉換').all()[$itemIndex]?.json
```

或在彙整節點使用：

```javascript
$('格式轉換').all().map(item => item.json)
```

### Code doesn't return items properly

錯誤：

```text
Code doesn't return items properly
```

原因：

```text
Code 節點沒有回傳 array of objects
```

處理：

```javascript
return [{
  json: {
    ...
  }
}];
```

## 後續開發注意事項

- 後續若新增其他品項主檔，應在主檔查詢階段加入品項分類或多主檔載入策略。
- 若 OCR 經常不穩，應優化 Gemini prompt，要求輸出固定 JSON。
- 若 Notion 主檔規格文字格式不一致，可增加相似度比對或關鍵字比對，不一定只做完全相等。
- 若 LINE 回覆內容過長，Flex Message 只顯示前 8 筆，完整資料以 Notion 為主。
- 若要避免重複寫入，可增加圖片 messageId 或批次 ID 欄位。
- 若要追蹤每次掃描批次，可新增 `掃描批次ID`、`LINE userId`、`圖片ID`、`OCR狀態` 欄位。
